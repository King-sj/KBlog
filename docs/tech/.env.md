# 使用 .env 文件管理环境变量

`.env` 是一个简单的键值对文件，用来管理应用的不同环境配置（开发、测试、生产）。它把敏感的配置信息从代码中分离出来，避免硬编码。

<!-- more -->

## 基本用法

在项目根目录创建 `.env` 文件：

```text
# .env
DATABASE_URL=postgresql://localhost:5432/mydb
API_KEY=sk-abc123
LOG_LEVEL=debug
PORT=3000
```

格式就是 `KEY=VALUE`，一行一个。用 `#` 写注释。

然后在代码中读取：

### Node.js

```javascript
// 安装 dotenv
// npm install dotenv

require('dotenv').config();

const dbUrl = process.env.DATABASE_URL;
const apiKey = process.env.API_KEY;
```

### Python

```python
# 安装 python-dotenv
# pip install python-dotenv

from dotenv import load_dotenv
import os

load_dotenv()

db_url = os.getenv("DATABASE_URL")
api_key = os.getenv("API_KEY")
```

### VSCode 插件

VSCode 也有 [DotENV](https://marketplace.visualstudio.com/items?itemName=mikestead.dotenv) 插件，提供语法高亮和自动补全。

## 多环境管理

实际项目往往需要在不同环境使用不同的配置：

```text
# .env.development
DATABASE_URL=postgresql://localhost:5432/dev_db
LOG_LEVEL=debug

# .env.production
DATABASE_URL=postgresql://prod-server:5432/prod_db
LOG_LEVEL=warn
```

加载时区分环境：

```javascript
const env = process.env.NODE_ENV || 'development';
require('dotenv').config({ path: `.env.${env}` });
```

## 安全隐患

### 不要把 .env 提交到 Git

这是 `.env` 最重要的规则：

```text
# .gitignore
.env
.env.*
!.env.example
```

提交一个 `.env.example` 作为模板，包含所有需要的变量名和默认值说明：

```text
# .env.example
# 数据库连接地址
DATABASE_URL=postgresql://localhost:5432/mydb
# API 密钥（必填，前往 https://example.com/settings 获取）
API_KEY=
# 日志级别：debug / info / warn / error
LOG_LEVEL=info
```

### Docker 环境

不要在 Dockerfile 中用 `COPY .env .` 或 `ENV` 硬编码密钥。用 docker-compose 或容器编排的 secret 管理：

```yaml
# docker-compose.yml
services:
  app:
    image: my-app
    env_file:
      - .env.production
```

或者用 `docker run --env-file .env.production ...`。

更安全的方式是用 Docker secrets（Swarm/k8s）或 Vault 等密钥管理服务。

### 前端项目的 .env

Vite / Create React App 等前端构建工具也支持 `.env`，但**所有以 `VITE_` / `REACT_APP_` 开头的变量会被打包进 JS bundle**。不要在前端 `.env` 中放任何敏感信息 —— 它们是公开的。

```text
# .env（前端 Vite 项目）
VITE_API_BASE_URL=https://api.example.com   # OK，这是公开的
VITE_SECRET_KEY=sk-xxx                       # 绝对不行！会被打包进 JS
```

## 总结

`.env` 的核心价值是**环境配置与代码分离**。记住三条原则：

1. `.env` 永远不提交到 Git
2. 提供 `.env.example` 作为配置模板
3. 前端项目不要在前端 `.env` 中放密钥

就这么简单 ~~不需要搞太复杂~~。
